# Codex Council

Codex Council - это локальный плагин для Codex, который позволяет запускать
небольшой "совет" из субагентов внутри одной рабочей среды. Совет может
обсуждать продуктовую идею, спорный документ, архитектурное решение, план
исследования или код. Код здесь не обязательный сценарий, а один из вариантов.

Идея вдохновлена репозиторием Andrej Karpathy
[karpathy/llm-council](https://github.com/karpathy/llm-council). Там несколько
моделей отвечают на один и тот же вопрос, затем оценивают ответы друг друга, а
финальный ответ собирает отдельная модель-председатель. Смысл простой: один
ответ часто выглядит убедительно, но несколько независимых взглядов лучше
ловят слабые места.

Мы перенесли эту механику внутрь Codex. Здесь не нужно поднимать отдельный
веб-интерфейс, подключать OpenRouter или заводить отдельный API-ключ для
совета. Codex остается средой выполнения, а плагин добавляет локальный слой
координации, через который агенты могут обмениваться состоянием, заметками и
артефактами.

## Как это устроено

Родительский агент создает council-сессию и назначает роли: architect,
reviewer, verifier, skeptic, security, chair и, если пользователь явно разрешил
правки, writer. У каждой роли своя зона ответственности. Один агент собирает
сильную позицию, другой ищет слабые места, третий проверяет выводы по доступным
источникам: тексту задачи, документам, локальным файлам, командам, тестам или
логам, если они вообще нужны для этой сессии.

Агенты общаются через локальный MCP-сервер с SQLite. Короткие сообщения идут
через MCP-инструменты. Длинные заметки, черновики ревью, транскрипты, источники
и доказательства сохраняются как артефакты в рабочей папке:

```text
<workspace>/.codex-council/
  council.sqlite
  artifacts/
  exports/
```

Это ближе к общей доске, чем к обычной схеме "родитель пересказывает все всем
субагентам". Родительский агент все еще управляет сессией, но не обязан тащить
в своем контексте каждую промежуточную мысль. Агенты оставляют короткие
сообщения, ссылаются на артефакты, подтверждают прочитанное, добавляют claims и
голосуют по решениям.

## Зачем это нужно

Council mode не нужен для каждой мелкой задачи. Для простого вопроса обычный
Codex-процесс быстрее и дешевле. Совет начинает иметь смысл там, где есть
неопределенность: нужно сравнить варианты, найти слабые места в плане, провести
read-only review, проверить спорные допущения или подготовить аккуратное
решение перед тем, как что-то менять.

Разделение на роли дает несколько практических плюсов:

- агенты могут не соглашаться друг с другом до финального ответа;
- проверка отделена от идеи, дизайна и реализации;
- длинные доказательства лежат в файлах, а не забивают контекст родителя;
- после обсуждения остаются транскрипты и артефакты;
- writer mode включается только явно, поэтому review и design-сессии по
  умолчанию остаются read-only.

Все работает локально. Плагин пишет состояние совета в выбранный workspace, а
свой код хранит в директории плагина. Сейчас backend сделан на SQLite, этого
достаточно для рабочего локального сценария. Протокол специально оставлен
простым: позже storage можно заменить на Postgres или другой общий backend, не
переписывая то, как агенты разговаривают с council.

## Что лежит в репозитории

```text
plugins/codex-council/
  .codex-plugin/plugin.json   plugin metadata
  .mcp.json                   MCP server registration
  mcp/council_server.py       local MCP server
  skills/                     Codex skills for discussion, review, design, write mode
  docs/                       protocol, roles, architecture, storage, troubleshooting
  tests/                      server tests
```

Плагин рассчитан на работу внутри Codex. Типичные запросы:

```text
Run a council discussion about this question.
Use a council to compare these options.
Review this plan with a council.
Use a council to design a decision path for this change.
Implement this with council writer mode.
```

После сессии файлы в `.codex-council/` остаются в workspace, пока пользователь
сам их не удалит. Эту папку стоит воспринимать как проектные данные: внутри
могут быть фрагменты документов, исходников, логи, заметки ревью и другой
локальный контекст.
